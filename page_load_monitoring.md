# Мониторинг скорости загрузки продукта

## Проблема и цель

### Проблема

Текущий способ замера скорости загрузки - ручной: ресёрчеры просят людей проверить продукт. Выборка мне кажется маленькая, нерепрезентативная.

При этом у нас есть слепая зона: PostHog стоит на продукте, но не видит время прохождения через цепочку редиректоров (Allow24 + зеркало). Те кто отвалился до продукта в данных вообще не появятся.

### Цель

Получить автоматический, непрерывный поток реальных данных о скорости загрузки продукта от реальных пользователей из РФ с разбивкой по гео, устройству и каналу входа

## Job Story (JTBD)

Когда я хочу понять насколько быстро загружается продукт у пользователей из разных регионов РФ, я хочу открыть дашборд в PostHog и видеть актуальные перцентили по скорости за последние 7 дней с разбивкой по гео и устройствам, чтобы принимать решения на основе данных, а не на доверии к выборке из 10 человек.

## Acceptance Criteria

- Событие `page_load_performance` отправляется при каждой загрузке страницы продукта.
- Событие содержит все перечисленные параметры, включая `redirect_time_ms`, `ttfb_ms`, `dom_ready_ms`, `full_load_ms`, `is_pwa`, `page`.
- В PostHog построен дашборд с 5 графиками из блока 3.
- Алерт в Slack срабатывает при P75 `full_load` выше 5000 ms за 24 часа.
- После 7 дней работы в дашборде видны данные минимум по 3 регионам РФ с выборкой не менее 1000 событий на регион.

## Риски и ограничения

### Что не покроет это решение

Пользователи, отвалившиеся на редиректорах до продукта, не попадут в данные. Для них нужен отдельный мониторинг на уровне самих редиректоров.

Данные PostHog будут показывать только выживших. Это означает, что реальные цифры по всей аудитории хуже, чем мы увидим в дашборде.

### Технические ограничения

- `performance.timing` устарел в пользу `PerformanceNavigationTiming`. Разработке следует проверить, какой API поддерживается в целевых браузерах аудитории, особенно в старых версиях Android WebView. Если нужна совместимость, использовать `performance.getEntriesByType('navigation')[0]` с fallback на `performance.timing`.
- Событие отправляется при каждой загрузке страницы. Это может создать большой объём данных в PostHog при высоком трафике. Если нужно снизить нагрузку, можно применить сэмплирование: отправлять в среднем каждое пятое событие с помощью условия `Math.random() < 0.2`.

## Описание решения

### Блок 1. Web Vitals через `performance.timing`

При загрузке каждой страницы продукта фронт отправляет в PostHog одно кастомное событие `page_load_performance` со следующими параметрами:

```javascript
window.addEventListener('load', () => {
  const t = window.performance.timing;

  posthog.capture('page_load_performance', {
    redirect_time_ms: t.redirectEnd - t.redirectStart,
    ttfb_ms: t.responseStart - t.navigationStart,
    dom_ready_ms: t.domContentLoadedEventEnd - t.navigationStart,
    full_load_ms: t.loadEventEnd - t.navigationStart,
    page: window.location.pathname,
  });
});
```

- `redirect_time_ms` — время на прохождение всей цепочки редиректоров до продукта. Это единственный способ захватить время редиректов из браузера без мониторинга на самих редиректорах.
- `ttfb_ms` — полный путь от клика до первого байта продукта, включая редиректы.
- `dom_ready_ms` — момент, когда страница стала визуально готова.
- `full_load_ms` — момент, когда загрузилось всё, включая изображения и скрипты.

### Блок 2. Обогащение контекстом

К событию автоматически добавляются стандартные свойства PostHog: `$geoip_city`, `$geoip_country_code`, `$device_type`, `$os`, `$browser`. Дополнительно нужно прокинуть:

```javascript
posthog.capture('page_load_performance', {
  // ...метрики выше
  is_pwa: window.matchMedia('(display-mode: standalone)').matches,
  utm_source: new URLSearchParams(window.location.search).get('utm_source'),
  entry_url: document.referrer,
});
```

### Блок 3. Дашборд в PostHog

После сбора данных строятся следующие графики:

| График | Метрика | Разбивка |
| --- | --- | --- |
| TTFB по регионам | Медиана `ttfb_ms` | `$geoip_city` |
| Full load по устройствам | P50/P75/P95 `full_load_ms` | `$device_type` + `$os` |
| Редиректы | Медиана `redirect_time_ms` | `$geoip_country_code` |
| Тренд по времени | P75 `full_load_ms` | По дням |
| PWA vs Web | P75 `full_load_ms` | `is_pwa` |

### Блок 4. Алертинг

Настроить в PostHog алерт: если P75 `full_load_ms` за последние 24 часа превышает 5 000 мс — отправить уведомление в Slack. Это даст возможность оперативно реагировать на деградацию без ручного мониторинга.
